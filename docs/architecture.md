# Архитектура установщика

Этот документ объясняет **как устроен репозиторий** — для тех, кто хочет понять, что делает AI-агент, прежде чем давать ему SSH-доступ к серверу.

## Общая картина

```
┌──────────────────┐     ┌──────────────────────┐
│  Пользователь    │────►│  Claude Code / Codex │
│  (VS Code)       │     │  (AI-установщик)     │
└──────────────────┘     └──────────┬───────────┘
                                    │ читает
                         ┌──────────▼───────────┐
                         │  Этот репозиторий:   │
                         │  CLAUDE.md/AGENTS.md │
                         │  RUNBOOK.md          │
                         │  scripts/*.sh        │
                         │  templates/*         │
                         └──────────┬───────────┘
                                    │ SSH
                         ┌──────────▼───────────┐
                         │  VPS пользователя    │
                         │  (любой провайдер,   │
                         │   Ubuntu 22+)        │
                         └──────────────────────┘
```

## Компоненты

### 1. `CLAUDE.md` / `AGENTS.md`

Файл-инструкция, который **автоматически** подхватывается AI-агентом при открытии репо.

- **Claude Code** ищет `CLAUDE.md`.
- **Codex CLI** ищет `AGENTS.md`.
- Содержимое почти идентично (95% совпадения).

Определяет:

- Роль агента (установщик OpenClaw, не generic coder).
- Правила поведения (сначала RUNBOOK, потом действия).
- Границы (можно / нельзя).
- Как обрабатывать ошибки.
- Когда спрашивать пользователя, когда действовать самостоятельно.

### 2. `PROMPT-claude-code.md` / `PROMPT-codex.md`

Готовые **промпты для копипасты** пользователем в AI-установщик. Это entry-point: пользователь копирует промпт → вставляет в Claude Code / Codex → AI начинает работу.

Промпт короткий (5-10 строк) и содержит:

- Инструкцию AI'у клонировать/скачать репо.
- Команду начать с чтения `RUNBOOK.md`.
- Несколько критичных safety-check'ов.

### 3. `RUNBOOK.md`

**Single source of truth** для процесса установки. Детальный план для AI-агента:

- Фазы установки (preflight → SSH hardening → install → configure → OAuth → Telegram → pairing → audit).
- Успешный критерий каждой фазы (как понять что этап пройден).
- Recovery-действия при типичных ошибках.
- Точки входа для пользователя (когда и что запросить).

**Если нужно поменять процесс установки** — меняем только RUNBOOK. Не scripts, не PROMPT, не README. Остальное опирается на RUNBOOK.

### 4. `scripts/`

Проверенные shell-скрипты, которые AI **запускает**, а не импровизирует. Принципы:

- **Идемпотентность** — запуск дважды даёт тот же результат.
- **Fail-fast** — `set -euo pipefail` в каждом скрипте.
- **Логирование** — вывод шагов в stdout с префиксами `[+]`, `[✓]`, `[!]`.
- **Без интерактивности** — все данные через параметры или env-переменные (AI-агент их задаёт).
- **Минимальные зависимости** — только инструменты, которые есть по умолчанию в Ubuntu.

**Почему скрипты, а не inline-команды в RUNBOOK:**

AI-агенты имеют склонность к галлюцинациям. Один классический пример — команда `openclaw init`, которой не существует (правильно — `openclaw setup` или `openclaw onboard`). Скрипты — детерминированный путь, исключающий такие ошибки.

Полный список скриптов — см. `scripts/README.md` (будет добавлен).

### 5. `templates/`

Шаблоны конфигурационных файлов с плейсхолдерами:

- **`openclaw.json.tpl`** — главный конфиг OpenClaw. Скрипт `configure-openclaw.sh` подставляет реальные значения (токены, пути, user IDs).
- **`systemd-unit.service.tpl`** — fallback systemd unit для случаев, когда `openclaw onboard --install-daemon` не сработал.
- **`soul/`** — готовые SOUL.md для разных use case, пользователь выбирает один.

### 6. `docs/`

Для тех, кто хочет вникнуть глубже:

- **`how-it-works.md`** — расширенное объяснение что делает установщик (то же что RUNBOOK, но человеческим языком).
- **`troubleshooting.md`** — резюме типичных проблем + ссылки на Notion-гайд.
- **`architecture.md`** (этот файл) — архитектура репо.

## Разделение ответственности: что делает AI, что делает пользователь

| Этап | Делает |
|------|--------|
| Купить VPS у любого провайдера | **Пользователь** (сайт хостинга AI не контролирует) |
| Передать агенту IP и пароль | **Пользователь** |
| SSH на сервер | **AI** (через Claude Code / Codex с SSH-инструментом) |
| Смена пароля root | **AI** (команда + проверка) |
| Создание non-root пользователя | **AI** (через `scripts/ssh-harden.sh`) |
| Настройка UFW, fail2ban | **AI** (скрипт) |
| Установка Node.js + OpenClaw | **AI** (скрипт) |
| OAuth к ChatGPT Plus/Pro | **Пользователь** (клик по ссылке в браузере с VPN) + AI читает результат |
| Создание Telegram-бота в BotFather | **Пользователь** (AI не может управлять Telegram-клиентом) |
| Вставка токена бота | **Пользователь** передаёт AI → AI подставляет в конфиг |
| Pairing (код в чате → терминал) | **Пользователь** + AI (код приходит в бот, пользователь диктует AI, AI одобряет в CLI) |
| Security audit | **AI** (скрипт + анализ вывода) |
| Первое сообщение боту | **Пользователь** |

## Потенциальные ошибки и как мы их предотвращаем

| Ошибка | Профилактика |
|--------|--------------|
| AI галлюцинирует команду | Скрипты в `scripts/`, AI только запускает |
| AI выходит за скоуп (трогает не тот сервер) | CLAUDE.md/AGENTS.md явно ограничивает домен действий |
| AI теряет контекст на длинном процессе | RUNBOOK структурирован с явными фазами + success criteria |
| Ошибка в скрипте ломает систему | `set -euo pipefail`, идемпотентность, rollback где возможно |
| Секреты утекают в логи | Скрипты маскируют пароли/токены (`****` в выводе) |
| Несовместимость версии OpenClaw | `preflight-check.sh` валидирует версию, скрипт установки пинит стабильную |

## Как обновлять репо

**Канонический процесс:**

1. Issue с описанием проблемы / предложения.
2. Исправление в `drafts/`-форке.
3. Тест на чистом VPS (Ubuntu 22.04 и 24.04).
4. PR с результатами теста.
5. Review → merge → релиз версии.

**Версионирование:** semver. Версия в `VERSION` файле (будет добавлен в Этапе 2). `1.0.0` — первый публичный релиз.
